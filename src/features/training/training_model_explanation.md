# Training & Competency Module: Data Model & Flow Explanation

This document provides a technical overview of the Training & Competency module to assist in backend development and API integration.

## 1. Entity Overview

### Compliance Event Type (`compliance_event_types`)
The "Training" module is a specialized subset of the broader "Compliance Calendar". All training sessions are stored as events of a specific type.
- **Identification**: Filtered where `name == "Training"`.
- **Fields**: `id`, `name`, `category`, `color`, `icon`, `defaultFrequency`.

### Compliance Event (`compliance_events`)
Represents a scheduled training session or a recurring training requirement.
- **eventTypeId**: FK to the Training event type.
- **title**: Name of the training module (e.g., "Safety Protocol").
- **dueDate**: Scheduled date for the training.
- **status**: `pending`, `completed`, `overdue`, `in-progress`.
- **assignedTo**: Either a specific staff name or the string `"All Staff"`.
- **givenBy**: The trainer or speaker.
- **recurrence**: `one-time`, `monthly`, `quarterly`, `yearly`.
- **notes**: Additional instructions or links.

### Training Attendance (`training_attendance`)
Maps specific staff members to training events and tracks their individual progress.
- **eventId**: FK to `compliance_events`.
- **staffId**: FK to the Staff member.
- **status**: Individual status (`pending`, `completed`).
- **completionDate**: Date when the staff member finished the training.
- **score**: (Optional) Quiz or assessment result.

## 2. Key Business Logic & Flows

### Scheduling a Training
When a training session is scheduled:
1. A record is created in `compliance_events`.
2. **Attendance Snapshotting**: 
    - If assigned to "All Staff", a `training_attendance` record is created for **every** active staff member at that moment.
    - If assigned to a specific person, a single `training_attendance` record is created.
3. **Recurrence Handling**: If a recurrence is selected (e.g., Yearly), the system auto-generates future instances (typically the next 3) of both the event and the attendance records to ensure long-term compliance visibility.

### Status Calculation
- **Stored Status**: The primary status is stored in the database.
- **Calculated 'Overdue'**: If `dueDate < today` AND `status != "completed"`, the system treats the record as **Overdue** regardless of the stored status value.

### Competency Matrix Generation
The matrix is a pivot-table view generated by:
- **Rows**: All active staff members.
- **Columns**: Unique training **Titles** (not IDs, as titles represent the "Subject").
- **Cells**: Shows the status of the **latest** instance of that specific training subject for that specific staff member.
- **Logic**: If no attendance record exists for a staff/subject pair, it displays as "N/A" (Not Required).

## 3. Recommended Backend API Structure

To support this module, the following API capabilities are recommended:

### Endpoints
- `GET /trainings`: List all training events with basic stats (Total, Upcoming, Overdue).
- `POST /trainings`: Create a new training event. Should handle the logic of creating sub-attendance records server-side.
- `GET /trainings/matrix`: A specialized endpoint that returns the aggregated data needed for the competency matrix to avoid heavy client-side filtering.
- `PATCH /trainings/attendance/{id}`: Update specific staff completion status and date.

### Data Model Logic
- **Snapshotting**: Ensure that once a training is scheduled for "All Staff", new staff added to the system later do not automatically get assigned to *past* training events, only future ones.
- **Auto-Generation**: Consider a cron job or worker to generate the next recurring instance of a training event once the current one is completed or its due date has passed.
