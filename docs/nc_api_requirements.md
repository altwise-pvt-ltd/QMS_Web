# Backend API Requirements: NC Categories & Subcategories

This document outlines the requirements for implementing dynamic Non-Conformance (NC) categories and subcategories in the backend to replace the current static frontend implementation.

## 1. Database Schema

The backend must implement two new tables with a parent-child relationship.

### NC_Categories (High-level Quality/Risk Indicators)
- `Id`: Integer (Primary Key, Identity/Auto-increment)
- `Name`: String (e.g., "Pre-Analytical", "Personnel")

### NC_Subcategories (Specific NC Types / Nature of NC)
- `Id`: Integer (Primary Key, Identity/Auto-increment)
- `CategoryId`: Integer (Foreign Key to `NC_Categories.Id`)
- `Name`: String (e.g., "Improper sample collection", "Obsolete SOP in use")

## 3. API Endpoints

### 3.1 Fetch Categories
**Endpoint:** `GET /api/NonConformance/GetCategories`
**Response:** `Array<Category>`
```json
[
  { "id": 1, "name": "Pre-Analytical" },
  { "id": 2, "name": "Analytical" }
]
```

### 3.2 Fetch Subcategories by Category
**Endpoint:** `GET /api/NonConformance/GetSubCategories/{categoryId}`
**Response:** `Array<Subcategory>`
```json
[
  { "id": 1, "name": "Improper sample collection" },
  { "id": 2, "name": "Patient misidentification" }
]
```

---

## 4. Integration with Existing NC Endpoints

### 4.1 Create/Update NC
The existing endpoints already accept `NonConformanceCategoryId` and `NonConformanceSubCategoryId` as integer IDs in `FormData`.
- **POST** `/api/NonConformance/CreateNonConformance`
- **PUT** `/api/NonConformance/UpdateNonConformance/{id}`

**Action Required:** Ensure these fields now act as formal Foreign Keys referencing the new `NC_Categories` and `NC_Subcategories` tables.

### 4.2 Fetch NC (All/Single)
The `GetAllNonConformances` and `GetNonConformanceById` endpoints must be updated to return the resolved **names** along with the IDs.
- **GET** `/api/NonConformance/GetAllNonConformances`
- **GET** `/api/NonConformance/GetNonConformanceById/{id}`

**Expected Response Improvement:**
```json
{
  "nonConformanceId": 123,
  "nonConformanceCategoryId": 1,
  "nonConformanceCategoryName": "Pre-Analytical", // New Field
  "nonConformanceSubCategoryId": 2,
  "nonConformanceSubCategoryName": "Patient misidentification", // New Field
  // ... other fields
}
```
*Note: Returning names avoids multiple lookups on the frontend.*

---

## 5. Implementation Notes
- **Direct ID Usage:** The IDs generated by the database will be used directly by the frontend.
- **Fixed Reference:** These categories are not user-editable via the UI at this stage.
- **Consistency:** Subcategories should accurately reflect the "Nature of NC" as documented in `NcCategories.js`.
